# 微信小游戏 2.5D 技术方案

本文档定义项目正式客户端方向：**Cocos Creator + 微信小游戏 + 2.5D 竖屏挂机挖宝**。现有 Web 原型、Mock API、CSV/JSON 配置继续保留，用于验证玩法、接口和数值。

## 1. 定位

```text
平台：微信小游戏
画面：2.5D 斜 45 度/伪 3D 场景
玩法：挂机挖宝、鉴宝估价、打怪推进、装备境界成长
方向：先做可试玩研发版，再考虑商业化 SDK
```

不建议把正式客户端做成普通微信小程序。普通小程序更适合工具和内容服务；本项目需要动画、场景、战斗、资源加载和游戏循环，应按微信小游戏研发。

## 2. 技术选型

| 模块 | 选择 | 原因 |
| --- | --- | --- |
| 引擎 | Cocos Creator | 适合 2.5D/2D 轻量游戏，微信小游戏适配成熟，包体压力较小 |
| 语言 | TypeScript | 和 Cocos Creator 工程生态匹配，适合 UI/API/状态管理 |
| 渲染 | 2D 节点 + 斜 45 度场景层 | 达到 2.5D 视觉，不承担真 3D 性能和包体成本 |
| 动画 | Spine/序列帧/骨骼动画 | 角色、怪物、挖宝动作、鉴宝揭晓 |
| 数据 | 继续使用 `Config/json/` | 和现有配置导出工具兼容 |
| 后端 | 继续使用 HTTP API | 先接 Mock API，后续替换正式服务 |

## 3. 2.5D 表现规则

### 3.1 主场景

主场景采用斜 45 度视觉：

- 地面为菱形或斜透视平台。
- 角色站在场景中下部，面向矿点或宝箱。
- 矿点、宝箱、古物从地面弹出。
- 场景层级按 `y` 坐标排序，避免角色和物件穿插错误。

推荐层级：

```text
Background
  -> FarDecor
  -> Ground
  -> ResourceNodes
  -> Character
  -> DropsAndTreasure
  -> NearDecor
  -> UI
```

### 3.2 地图风格

| 地图 | 2.5D 表现 |
| --- | --- |
| 新手矿洞 | 暖色矿道、碎石、矿车、浅层宝箱 |
| 灵脉山谷 | 灵气水流、草木、发光矿脉 |
| 古修遗迹 | 石碑、残墙、机关、古物碎片 |
| 幽冥秘境 | 冷色雾气、骨灯、裂缝、阴影怪物 |
| 仙界裂隙 | 浮空石、星光裂缝、高阶宝箱 |

### 3.3 角色与怪物

- 角色使用 2D Spine 或序列帧，表现待机、挖掘、攻击、受击、胜利。
- 怪物使用 2D Spine/序列帧，表现待机、攻击、受击、死亡。
- 不做真 3D 模型，避免微信小游戏包体和性能压力。

## 4. 核心界面

### 4.1 主界面

首屏直接是可操作场景，不做空城镇。

| 区域 | 内容 |
| --- | --- |
| 顶部栏 | 境界、战力、灵石、玉、鉴宝符 |
| 中央 2.5D 场景 | 角色自动挖宝、矿点、宝箱、掉落反馈 |
| 主线条 | 当前目标、奖励、可领取状态 |
| 右侧入口 | 鉴宝、打怪、装备、背包 |
| 底部导航 | 挖宝、角色、装备、任务、商店占位 |

### 4.2 鉴宝估价

鉴宝是主要爽点，使用独立 UI 层：

- 古物居中展示。
- 未鉴定时显示剪影/灰尘/封印。
- 点击鉴定后播放擦拭、光效、价格跳字。
- 结果分普通、小赚、大赚、极品四档。
- MVP 不接广告，不弹激励视频。

### 4.3 打怪关卡

打怪可使用 2.5D 小战斗场：

- 我方在左下，怪物在右上或右侧。
- 自动攻击，不需要玩家操作技能。
- 展示推荐战力、首通奖励、战斗结果。
- 服务端最终判定胜负和奖励。

## 5. Cocos 工程建议

目录建议：

```text
Client/wechat-minigame/
  assets/
    scenes/
      Boot.scene
      Home.scene
      Battle.scene
      Appraisal.scene
    scripts/
      api/
      config/
      state/
      ui/
      gameplay/
      utils/
    resources/
      config/
      sprites/
      spine/
      audio/
  settings/
```

核心模块：

| 模块 | 职责 |
| --- | --- |
| `api/ApiClient.ts` | HTTP 请求、错误处理、token |
| `config/ConfigManager.ts` | 加载 JSON 配置 |
| `state/GameState.ts` | 玩家状态、资产、任务、引导 |
| `gameplay/IdleDigController.ts` | 挖宝动画和收益展示 |
| `gameplay/BattleController.ts` | 自动战斗表现 |
| `ui/HomeView.ts` | 主界面 |
| `ui/AppraisalView.ts` | 鉴宝估价 |
| `ui/QuestBar.ts` | 主线条 |

## 6. 与现有系统对接

继续复用现有接口：

| 行为 | 接口 |
| --- | --- |
| 登录 | `/auth/guest-login` |
| 同步 | `/game/sync` |
| 主线领取 | `/quest/claim` |
| 引导完成 | `/tutorial/complete` |
| 挂机领取 | `/idle/claim` |
| 鉴宝 | `/antique/appraise` |
| 战斗 | `/battle/start`、`/battle/finish` |
| 装备 | `/equipment/equip` |
| 境界 | `/realm/breakthrough` |

Mock API 仍作为研发期后端。正式后端上线前，客户端不直接相信本地计算的资产、战斗和任务奖励。

## 7. 包体与加载

微信小游戏需要控制首包：

- 首包只放启动页、主界面基础 UI、新手地图、首批角色动画。
- 后续地图、怪物、鉴宝特效、音频走分包或远程资源。
- 所有图集按界面/地图拆分，不做巨大公共图集。
- Spine/序列帧资源必须限制帧数和尺寸。
- 音频先只保留点击、挖宝、鉴宝、战斗胜利四类基础音效。

## 8. MVP 研发顺序

1. 建 Cocos Creator 微信小游戏工程骨架。
2. 接入本地 JSON 配置加载。
3. 接入 Mock API：登录、同步、重置。
4. 做 2.5D 主场景：角色待机、挖宝循环、资源栏。
5. 接主线条和任务可领取状态。
6. 做鉴宝估价界面。
7. 做打怪关卡界面。
8. 做装备穿戴和境界突破。
9. 做微信小游戏真机预览和包体检查。

## 9. 暂缓项

- 微信支付。
- 激励广告。
- 排行榜。
- 分享裂变。
- 正式活动系统。
- 大量美术皮肤。

这些等核心 2.5D 玩法闭环稳定后再接。
