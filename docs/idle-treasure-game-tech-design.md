# 挂机放置挖宝游戏技术文档

> 项目定位：面向商业化发行的挂机放置类挖宝游戏，玩法参考“道友来挖宝”一类的轻量成长、离线收益、装备/法宝收集与长期数值追求体验。本文档用于指导立项、原型、研发排期、后端架构、数据配置、商业化与上线运维。

## 1. 产品目标

### 1.1 核心体验

- **轻操作**：玩家通过派遣角色、升级工具、探索地图获得资源，日常操作控制在 3-10 分钟。
- **强成长**：围绕境界、装备、法宝、宠物/灵兽、洞府、图鉴等系统形成长期目标。
- **可离线**：支持离线收益结算，让玩家即使不在线也能持续获得基础资源。
- **高随机**：挖宝掉落、秘境事件、词条装备与宝箱奖励提供惊喜感。
- **可商业化**：通过月卡、战令、礼包、广告激励、皮肤外观、加速道具等形成多层付费点。

### 1.2 商业化目标

- D1 留存目标：35%-45%。
- D7 留存目标：12%-20%。
- 付费转化目标：3%-8%。
- 广告变现目标：激励视频每日 3-8 次自然触达。
- 首月重点：验证核心循环、离线收益、掉落爽感与礼包转化。

## 2. 玩法概述

### 2.1 核心循环

```text
上线领取离线收益
  -> 挖宝/探索获得资源、装备、材料
  -> 升级角色/工具/洞府/法宝
  -> 解锁更高地图与更稀有掉落
  -> 完成任务、图鉴、活动
  -> 继续挂机积累
```

### 2.2 主要系统

| 系统 | 目的 | 核心产出 |
| --- | --- | --- |
| 挖宝挂机 | 游戏主循环 | 灵石、矿石、装备、碎片、宝箱 |
| 地图秘境 | 阶段目标与掉落分层 | 高级材料、稀有法宝、事件奖励 |
| 角色境界 | 长线成长 | 属性、地图解锁、功能解锁 |
| 装备词条 | 随机追求 | 战力、效率、套装效果 |
| 法宝/灵兽 | 收集与养成 | 特殊技能、挂机加成 |
| 洞府建设 | 资源消耗与产出 | 被动收益、炼丹炼器 |
| 任务成就 | 引导与目标 | 货币、经验、抽奖券 |
| 活动运营 | 留存与收入 | 限时道具、排行榜奖励 |

## 3. 技术选型建议

### 3.1 客户端

优先建议使用 **Unity 2022 LTS/Unity 6 LTS**：

- 适合 2D 放置、UI 驱动、跨平台 Android/iOS/PC。
- 商业化 SDK、广告 SDK、数据埋点 SDK 接入成熟。
- 支持 Addressables 做资源热更新与分包加载。

可选方案：

- **Cocos Creator**：适合 H5/小游戏平台，包体轻，前端团队上手快。
- **Godot**：开源低成本，但商业 SDK 生态与移动端变现接入需要更多验证。

### 3.2 后端

建议采用轻量服务端，避免纯单机带来的作弊与运营能力不足。

| 模块 | 推荐方案 |
| --- | --- |
| API 服务 | Node.js/NestJS 或 Go/Gin |
| 数据库 | PostgreSQL/MySQL |
| 缓存 | Redis |
| 配置管理 | JSON/CSV + 后台发布 + 客户端版本校验 |
| 日志分析 | ClickHouse/BigQuery/云厂商日志服务 |
| 部署 | Docker + 云服务器/容器服务 |

### 3.3 账号与存档

- 支持游客账号、平台账号、手机号/邮箱绑定。
- 服务端保存关键资产：货币、付费道具、装备、活动进度、邮件。
- 客户端可缓存展示数据，但服务端需校验收益与资产变更。
- 存档结构需要带 `schema_version`，方便后续迁移。

## 4. 系统设计

### 4.1 挂机收益结算

#### 4.1.1 基础公式

```text
有效离线时长 = min(当前时间 - 上次结算时间, 离线收益上限)
基础收益 = 地图基础产出 × 挖掘效率 × 有效离线时长
最终收益 = 基础收益 × VIP/月卡加成 × 活动加成 × 装备法宝加成
```

#### 4.1.2 结算原则

- 服务端以可信时间为准，客户端只做预览。
- 离线收益上限初始 8 小时，可通过月卡/特权提升至 12-24 小时。
- 稀有掉落使用服务端随机种子或服务端开奖，避免本地篡改。
- 收益包分为“确定收益”和“概率掉落”，便于展示与追踪。

### 4.2 掉落系统

掉落表建议拆分：

```text
DropGroup
  - group_id
  - map_id
  - unlock_condition
  - entries[]

DropEntry
  - item_id
  - weight
  - min_count
  - max_count
  - pity_rule_id
  - daily_limit
```

设计要点：

- 普通资源高频掉落，保证正反馈。
- 稀有装备/法宝使用保底机制，降低挫败感。
- 地图越高级，掉落池越聚焦，减少低价值奖励污染。
- 重要商业化道具需限制免费产出速度，避免经济系统失控。

### 4.3 装备与词条

装备结构：

```json
{
  "uid": "eq_123456",
  "template_id": "sword_001",
  "quality": "legendary",
  "level": 12,
  "main_attr": { "atk": 120 },
  "sub_attrs": [
    { "type": "dig_speed", "value": 0.08 },
    { "type": "crit_drop", "value": 0.03 }
  ],
  "created_at": 1760000000
}
```

规则建议：

- 品质：凡品、良品、珍品、灵品、仙品、神品。
- 词条分为战斗类、挖宝效率类、掉落类、商业化体验类。
- 高品质装备应可回收为强化材料，避免背包堆积。
- 装备洗练需要消耗稀缺材料，可作为中后期付费点。

### 4.4 地图与秘境

地图分层：

1. 新手矿洞：基础资源、快速升级。
2. 灵脉山谷：材料掉落、法宝碎片。
3. 古修遗迹：事件、宝箱、稀有装备。
4. 幽冥秘境：高阶材料、限时入场券。
5. 仙界裂隙：赛季活动与排行榜。

秘境事件类型：

- 选择事件：消耗资源换奖励。
- 战斗事件：根据战力/属性判定。
- 宝箱事件：看广告或消耗钥匙翻倍。
- 商人事件：刷新限购礼包或稀有材料。

## 5. 商业化设计

### 5.1 付费点

| 付费项 | 内容 | 注意事项 |
| --- | --- | --- |
| 首充礼包 | 高性价比法宝/外观/货币 | 不应破坏数值平衡 |
| 月卡 | 每日货币、离线时长、广告跳过券 | 核心长期转化点 |
| 战令 | 任务成长线、赛季奖励 | 免费/付费双轨 |
| 限时礼包 | 境界突破、地图解锁、活动冲榜 | 结合玩家节点触发 |
| 激励广告 | 收益翻倍、快速挖掘、免费刷新 | 控制频率，避免疲劳 |
| 外观皮肤 | 角色、铲子、洞府、灵兽 | 尽量不影响平衡 |

### 5.2 广告触点

- 离线收益 2 倍领取。
- 每日免费高级宝箱。
- 秘境失败复活/继续探索。
- 商店免费刷新。
- 快速挖掘 30-120 分钟收益。

### 5.3 防付费过强策略

- 付费提升效率，不直接跳过全部成长。
- 排行榜按赛季分组，控制大 R 对普通玩家压迫。
- 外观与便利性付费占比逐步提高。

## 6. 数据配置规范

建议所有核心数值配置化：

```text
/config
  items.csv
  maps.csv
  drops.csv
  equipment.csv
  skills.csv
  quests.csv
  monetization.csv
  events.csv
```

配置字段应包含：

- `id`：稳定唯一 ID，不随名称变化。
- `name_key`：多语言 key。
- `unlock_condition`：功能解锁条件。
- `start_time/end_time`：活动类配置。
- `version`：配置版本。

发布流程：

1. 策划修改表格。
2. 自动校验字段、引用、数值范围。
3. 导出 JSON。
4. 上传后台并生成版本号。
5. 客户端启动时拉取版本差异。

## 7. 反作弊与安全

必须服务端校验：

- 付费订单与发货。
- 离线收益时间与上限。
- 稀有掉落结果。
- 排行榜积分。
- 活动奖励领取。

基础策略：

- API 请求签名：时间戳 + nonce + token。
- 关键资产流水不可变更，只追加日志。
- 异常检测：离线收益过高、广告次数异常、地图进度跳跃。
- 订单幂等：同一平台订单号只能发货一次。

## 9. 研发里程碑

### 9.1 原型期：2-3 周

- 完成核心挖宝循环。
- 完成离线收益结算。
- 完成 3 张地图、20 个道具、基础装备。
- 完成基础 UI 与本地配置读取。

### 9.2 Alpha：4-6 周

- 接入账号、存档、服务端资产。
- 完成掉落、任务、背包、装备、境界系统。
- 接入埋点与错误日志。
- 完成第一版数值曲线。

### 9.3 Beta：4-6 周

- 接入广告、内购、月卡、战令。
- 完成活动、邮件、公告、排行榜。
- 做首轮留存与付费调优。
- 完成美术、音效、性能优化。

### 9.4 上线准备：2-4 周

- 完成隐私政策、用户协议、版号/平台资料准备。
- 压测、灰度、热更验证。
- 渠道 SDK 与支付回调验收。
- 准备首月运营活动。

## 10. MVP 研发范围

MVP 目标不是做完整商业版本，而是在 6-10 周内完成一个可测试、可调数值、可接广告和付费的闭环版本。首版重点验证三件事：

1. 玩家是否愿意每天回来领取收益、升级、解锁地图。
2. 掉落、装备、境界突破是否有足够爽感和长期追求。
3. 激励广告、月卡、礼包是否能自然嵌入核心循环。

### 10.1 MVP 必做功能

| 模块 | 范围 | 验收标准 |
| --- | --- | --- |
| 账号登录 | 游客登录、设备绑定、服务端 user_id | 卸载重装后绑定账号可恢复关键存档 |
| 挂机挖宝 | 在线/离线收益、地图产出、收益领取 | 离线 1 分钟到 8 小时收益正确，超过上限不继续累计 |
| 地图系统 | 5 张地图、地图解锁、地图掉落池 | 玩家能按境界/任务推进解锁新地图 |
| 角色境界 | 10 个境界、突破消耗、属性加成 | 境界升级影响挖宝效率和地图准入 |
| 背包道具 | 资源、材料、宝箱、碎片 | 道具增减有服务端流水，前端显示一致 |
| 装备系统 | 6 部位、6 品质、主词条/副词条、回收 | 装备可穿戴、替换、回收，战力/效率实时刷新 |
| 任务系统 | 主线任务、日常任务、成就 | 主线可引导前 30 分钟体验，日常每日重置 |
| 广告激励 | 离线收益翻倍、免费宝箱、快速挖宝 | 广告成功后发奖，失败/取消不发奖 |
| 月卡 | 购买、每日领取、离线上限提升 | 支付成功后权益立即生效，跨天可领取每日奖励 |
| 邮件 | 系统邮件、补偿邮件、附件领取 | 附件只能领取一次，领取结果可追踪 |
| 配置热更 | 拉取版本、校验配置、灰度开关 | 不发包可调整掉落、任务、礼包和活动参数 |
| 埋点日志 | 新手漏斗、经济产消、广告、付费 | 能看 D1 留存漏斗和主要资源库存 |

### 10.2 MVP 暂缓功能

| 功能 | 暂缓原因 | 后续时机 |
| --- | --- | --- |
| PVP / 竞技场 | 数值平衡复杂，容易拖慢首版验证 | 核心留存稳定后 |
| 公会/宗门 | 需要社交、聊天、贡献、排行配套 | Beta 后期 |
| 玩家交易 | 风控和经济系统压力大 | 不建议早期做 |
| 深剧情分支 | 内容成本高，对核心验证帮助有限 | 美术和世界观定型后 |
| 多角色阵容 | 会放大养成复杂度 | 装备/法宝循环验证后 |
| 大型赛季活动 | 依赖运营后台和内容产能 | 上线前 1-2 个月 |

### 10.3 首版内容量

| 内容 | 数量 | 说明 |
| --- | --- | --- |
| 境界 | 10 阶 | 每阶 3-5 个小目标，突破消耗逐步增加 |
| 地图 | 5 张 | 新手矿洞、灵脉山谷、古修遗迹、幽冥秘境、仙界裂隙 |
| 道具 | 80 个以内 | 资源、材料、宝箱、钥匙、装备碎片、广告券 |
| 装备 | 6 部位 × 6 品质 | 武器、衣服、护符、戒指、鞋子、法器 |
| 任务 | 主线 40 条、日常 8 条、成就 30 条 | 主线覆盖前 30 分钟 |
| 礼包 | 首充、月卡、成长礼包、限时礼包 | MVP 不做复杂订阅体系 |
| 广告点 | 3-5 个 | 控制频率，优先验证高价值触点 |

### 10.4 前 30 分钟体验脚本

| 时间段 | 玩家行为 | 系统目标 |
| --- | --- | --- |
| 0-3 分钟 | 登录、领取新手铲、第一次挖宝 | 建立“挖就有收益”的即时反馈 |
| 3-8 分钟 | 升级工具、穿第一件装备 | 让玩家理解效率提升 |
| 8-15 分钟 | 解锁第二张地图、获得蓝色装备 | 制造明显阶段变化 |
| 15-20 分钟 | 第一次境界突破 | 给出长期成长主线 |
| 20-25 分钟 | 触发离线收益教学或快速挖宝 | 引导广告翻倍但不强迫 |
| 25-30 分钟 | 开启日常任务、展示月卡权益 | 引导次日回访和轻付费 |

## 11. 技术架构落地

### 11.1 总体架构

```text
Unity Client
  -> API Gateway
  -> Game API Service
      -> PostgreSQL/MySQL
      -> Redis
      -> Config Service
      -> Payment Callback Service
      -> Analytics/Event Collector
  -> Admin Console
```

MVP 阶段可以先做单体后端，但模块边界要清晰，避免后续拆分困难。推荐把账号、资产、挂机、掉落、订单、配置、邮件、埋点按目录隔离。

### 11.2 客户端模块

| 模块 | 职责 |
| --- | --- |
| `Core` | 启动流程、状态机、时间同步、错误处理 |
| `Config` | 配置下载、版本校验、本地缓存、配置解析 |
| `Network` | API 封装、重试、鉴权、错误码处理 |
| `Account` | 游客登录、绑定、token 管理 |
| `Inventory` | 背包展示、资产变更提示、道具详情 |
| `Digging` | 当前地图、挂机状态、收益预览、领取表现 |
| `Equipment` | 装备列表、穿戴、替换、回收、战力刷新 |
| `Quest` | 主线、日常、成就进度和奖励领取 |
| `Shop` | 月卡、礼包、广告入口、订单状态 |
| `UI` | 页面栈、弹窗队列、红点、通用列表 |
| `Analytics` | 埋点队列、失败缓存、批量上报 |

客户端原则：

- 关键资产以服务端返回为准，客户端只做表现和缓存。
- 每次进入主界面后请求一次 `sync`，修正配置、服务器时间、资产摘要。
- UI 页面不直接改资产，统一走本地状态仓库和服务端回包刷新。
- 弱网下允许重试领取，但服务端必须幂等。

### 11.3 后端模块

| 模块 | 职责 |
| --- | --- |
| Account | 创建游客、绑定账号、token 签发、封禁状态 |
| Profile | 玩家基础信息、境界、地图进度、新手状态 |
| Asset | 货币、道具、装备、资产流水 |
| IdleReward | 在线/离线收益结算、广告翻倍、快速挖宝 |
| Drop | 掉落表解析、权重随机、保底计数 |
| Quest | 任务进度、日常重置、奖励领取 |
| Monetization | 商品、订单、月卡权益、礼包购买限制 |
| Mail | 邮件发送、附件领取、过期清理 |
| Config | 配置版本、发布、灰度、回滚 |
| Admin | 玩家查询、补偿发放、配置发布、公告 |
| Analytics | 事件收集、核心指标入库或转发 |

### 11.4 推荐 API 列表

| 接口 | 方法 | 用途 |

### 12.3 资产流水规则

- 所有货币、材料、装备增减都必须写入流水。
- `request_id` 必须唯一，避免网络重试造成重复发奖。
- 流水 `reason` 使用枚举，例如 `idle_claim`、`quest_reward`、`equipment_recycle`、`iap_purchase`、`admin_mail`。
- 支付、广告、邮件附件这类奖励必须先校验状态，再发奖，再更新领取状态。

### 13.2 数值曲线原则

- 前 10 分钟每 1-2 分钟给一次明显成长。
- 前 30 分钟至少解锁 2 张地图、完成 1 次境界突破、获得 1 件高品质装备。
- 第 1 天目标时长 20-40 分钟，离线回流收益要足够推动下一次升级。
- 第 3 天开始放慢境界速度，把追求转向装备词条、法宝碎片和日常活动。
- 免费玩家每天可完成 70%-80% 的基础成长，广告和月卡提供效率提升。

### 13.3 掉落模拟要求

研发期需要提供一个简单脚本或后台工具，用于模拟：

- 单地图挂机 1 小时、8 小时、24 小时的资源产出。
- 稀有装备 1000 次、10000 次掉落分布。
- 加成叠加后普通玩家、广告玩家、月卡玩家的成长差距。
- 每日资源产出与核心消耗是否平衡。

## 14. 后台与运营工具

MVP 后台不追求复杂，但必须覆盖上线后最容易出问题的操作。

| 功能 | MVP 范围 |
| --- | --- |
| 玩家查询 | 根据 user_id 查询账号、资产、订单、邮件、关键进度 |
| 资产补发 | 通过邮件发放补偿，禁止直接改资产余额 |
| 配置发布 | 上传 JSON、校验 hash、灰度、回滚 |
| 商品管理 | 查看商品配置、上下架开关、购买限制 |
| 邮件系统 | 全服邮件、指定玩家邮件、附件模板 |
| 公告系统 | 登录公告、维护公告、活动公告 |
| 订单查询 | 根据平台订单号查询发货状态和补单 |
| 风控标记 | 封禁、解除封禁、异常玩家备注 |

后台操作要求：

- 所有后台操作写审计日志。
- 高风险操作需要二次确认。
- 配置发布前必须自动校验引用关系和数值范围。
- 补偿邮件必须记录操作人、原因、附件内容。

## 18. 风险清单

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 数值膨胀过快 | 生命周期缩短 | 做经济模拟与每日产消看板 |
| 广告频率过高 | 留存下降 | 只在高价值节点触达 |
| 纯本地存档作弊 | 排行榜与付费受损 | 服务端校验关键资产 |
| 内容消耗过快 | 中后期流失 | 周活动、赛季、装备词条追求 |
| 付费强度过高 | 口碑下降 | 控制付费效率上限，增加外观付费 |
| 后端边界不清 | 后续接支付、活动、风控困难 | MVP 就建立资产流水和幂等规则 |
| 配置不可回滚 | 错误数值可能导致线上事故 | 配置发布必须支持校验、灰度、回滚 |
| 首日体验太慢 | 新用户无法感知成长爽感 | 前 30 分钟设定密集目标和高反馈奖励 |
