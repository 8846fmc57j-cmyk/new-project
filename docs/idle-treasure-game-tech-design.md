# 挂机放置挖宝游戏技术文档

> 项目定位：面向商业化发行的挂机放置类挖宝游戏，玩法参考“道友来挖宝”一类的轻量成长、离线收益、装备/法宝收集与长期数值追求体验。本文档用于指导立项、原型、研发排期、后端架构、数据配置、商业化与上线运维。

## 1. 产品目标

### 1.1 核心体验

- **轻操作**：玩家通过派遣角色、升级工具、探索地图获得资源，日常操作控制在 3-10 分钟。
- **强成长**：围绕境界、装备、法宝、宠物/灵兽、洞府、图鉴等系统形成长期目标。
- **可离线**：支持离线收益结算，让玩家即使不在线也能持续获得基础资源。
- **高随机**：挖宝掉落、秘境事件、词条装备与宝箱奖励提供惊喜感。
- **可商业化**：MVP 先以月卡、礼包、皮肤外观、加速道具等为主，广告激励暂缓到后续版本。

### 1.2 商业化目标

- D1 留存目标：35%-45%。
- D7 留存目标：12%-20%。
- 付费转化目标：3%-8%。
- MVP 阶段暂不接广告 SDK，先验证核心留存、成长爽感和轻付费转化。
- 首月重点：验证核心循环、离线收益、掉落爽感与礼包转化。

## 2. 玩法概述

补充文档：

- [MVP 客户端界面与接口对接文档](mvp-client-screen-flow.md)

### 2.1 核心循环

```text
上线领取离线收益
  -> 挖宝/探索获得资源、装备、材料
  -> 升级角色/工具/洞府/法宝
  -> 解锁更高地图与更稀有掉落
  -> 完成任务、图鉴、活动
  -> 继续挂机积累
```

### 2.2 主要系统

| 系统 | 目的 | 核心产出 |
| --- | --- | --- |
| 挖宝挂机 | 游戏主循环 | 灵石、矿石、装备、碎片、宝箱 |
| 地图秘境 | 阶段目标与掉落分层 | 高级材料、稀有法宝、事件奖励 |
| 角色境界 | 长线成长 | 属性、地图解锁、功能解锁 |
| 装备词条 | 随机追求 | 战力、效率、套装效果 |
| 法宝/灵兽 | 收集与养成 | 特殊技能、挂机加成 |
| 洞府建设 | 资源消耗与产出 | 被动收益、炼丹炼器 |
| 任务成就 | 引导与目标 | 货币、经验、抽奖券 |
| 活动运营 | 留存与收入 | 限时道具、排行榜奖励 |

## 3. 技术选型建议

### 3.1 客户端

优先建议使用 **Unity 2022 LTS/Unity 6 LTS**：

- 适合 2D 放置、UI 驱动、跨平台 Android/iOS/PC。
- 商业化 SDK、数据埋点 SDK 接入成熟，后续接广告 SDK 也有成熟生态。
- 支持 Addressables 做资源热更新与分包加载。

可选方案：

- **Cocos Creator**：适合 H5/小游戏平台，包体轻，前端团队上手快。
- **Godot**：开源低成本，但商业 SDK 生态与移动端变现接入需要更多验证。

### 3.2 后端

建议采用轻量服务端，避免纯单机带来的作弊与运营能力不足。

| 模块 | 推荐方案 |
| --- | --- |
| API 服务 | Node.js/NestJS 或 Go/Gin |
| 数据库 | PostgreSQL/MySQL |
| 缓存 | Redis |
| 配置管理 | JSON/CSV + 后台发布 + 客户端版本校验 |
| 日志分析 | ClickHouse/BigQuery/云厂商日志服务 |
| 部署 | Docker + 云服务器/容器服务 |

### 3.3 账号与存档

- 支持游客账号、平台账号、手机号/邮箱绑定。
- 服务端保存关键资产：货币、付费道具、装备、活动进度、邮件。
- 客户端可缓存展示数据，但服务端需校验收益与资产变更。
- 存档结构需要带 `schema_version`，方便后续迁移。

## 4. 系统设计

### 4.1 挂机收益结算

#### 4.1.1 基础公式

```text
有效离线时长 = min(当前时间 - 上次结算时间, 离线收益上限)
基础收益 = 地图基础产出 × 挖掘效率 × 有效离线时长
最终收益 = 基础收益 × VIP/月卡加成 × 活动加成 × 装备法宝加成
```

#### 4.1.2 结算原则

- 服务端以可信时间为准，客户端只做预览。
- 离线收益上限初始 8 小时，可通过月卡/特权提升至 12-24 小时。
- 稀有掉落使用服务端随机种子或服务端开奖，避免本地篡改。
- 收益包分为“确定收益”和“概率掉落”，便于展示与追踪。

### 4.2 掉落系统

掉落表建议拆分：

```text
DropGroup
  - group_id
  - map_id
  - unlock_condition
  - entries[]

DropEntry
  - item_id
  - weight
  - min_count
  - max_count
  - pity_rule_id
  - daily_limit
```

设计要点：

- 普通资源高频掉落，保证正反馈。
- 稀有装备/法宝使用保底机制，降低挫败感。
- 地图越高级，掉落池越聚焦，减少低价值奖励污染。
- 重要商业化道具需限制免费产出速度，避免经济系统失控。

### 4.3 装备与词条

装备结构：

```json
{
  "uid": "eq_123456",
  "template_id": "sword_001",
  "quality": "legendary",
  "level": 12,
  "main_attr": { "atk": 120 },
  "sub_attrs": [
    { "type": "dig_speed", "value": 0.08 },
    { "type": "crit_drop", "value": 0.03 }
  ],
  "created_at": 1760000000
}
```

规则建议：

- 品质：凡品、良品、珍品、灵品、仙品、神品。
- 词条分为战斗类、挖宝效率类、掉落类、商业化体验类。
- 高品质装备应可回收为强化材料，避免背包堆积。
- 装备洗练需要消耗稀缺材料，可作为中后期付费点。

### 4.4 地图与秘境

地图分层：

1. 新手矿洞：基础资源、快速升级。
2. 灵脉山谷：材料掉落、法宝碎片。
3. 古修遗迹：事件、宝箱、稀有装备。
4. 幽冥秘境：高阶材料、限时入场券。
5. 仙界裂隙：赛季活动与排行榜。

秘境事件类型：

- 选择事件：消耗资源换奖励。
- 战斗事件：根据战力/属性判定。
- 宝箱事件：消耗钥匙、鉴宝符或特殊道具提升奖励。
- 商人事件：刷新限购礼包或稀有材料。

## 5. 商业化设计

### 5.1 付费点

| 付费项 | 内容 | 注意事项 |
| --- | --- | --- |
| 首充礼包 | 高性价比法宝/外观/货币 | 不应破坏数值平衡 |
| 月卡 | 每日货币、离线时长、专属鉴宝次数 | 核心长期转化点 |
| 战令 | 任务成长线、赛季奖励 | 免费/付费双轨 |
| 限时礼包 | 境界突破、地图解锁、活动冲榜 | 结合玩家节点触发 |
| 外观皮肤 | 角色、铲子、洞府、灵兽 | 尽量不影响平衡 |

### 5.2 MVP 暂不接广告

MVP 阶段不设计激励视频入口，不接广告 SDK，不把广告奖励纳入核心经济。这样可以先集中验证挖宝、鉴宝估价、打怪、装备和境界成长是否成立。

后续如果数据证明核心循环稳定，再考虑增加广告点：

- 离线收益额外领取。
- 每日免费高级宝箱。
- 秘境失败继续探索。
- 商店免费刷新。
- 快速挖掘收益。

### 5.3 防付费过强策略

- 付费提升效率，不直接跳过全部成长。
- 排行榜按赛季分组，控制大 R 对普通玩家压迫。
- 外观与便利性付费占比逐步提高。

## 6. 数据配置规范

建议所有核心数值配置化：

```text
/config
  items.csv
  maps.csv
  drops.csv
  equipment.csv
  skills.csv
  quests.csv
  monetization.csv
  events.csv
```

配置字段应包含：

- `id`：稳定唯一 ID，不随名称变化。
- `name_key`：多语言 key。
- `unlock_condition`：功能解锁条件。
- `start_time/end_time`：活动类配置。
- `version`：配置版本。

发布流程：

1. 策划修改表格。
2. 自动校验字段、引用、数值范围。
3. 导出 JSON。
4. 上传后台并生成版本号。
5. 客户端启动时拉取版本差异。

## 7. 反作弊与安全

必须服务端校验：

- 付费订单与发货。
- 离线收益时间与上限。
- 稀有掉落结果。
- 排行榜积分。
- 活动奖励领取。

基础策略：

- API 请求签名：时间戳 + nonce + token。
- 关键资产流水不可变更，只追加日志。
- 异常检测：离线收益过高、地图进度跳跃、资产增长异常、战斗结算异常。
- 订单幂等：同一平台订单号只能发货一次。

## 10. 鉴宝估价与打怪玩法设计

这一块是 MVP 的核心差异化体验。挖宝不只给资源，而是产出“未知古物”和“怪物遭遇”，让玩家在短时间内获得判断、惊喜和成长反馈。

### 10.1 核心循环

```text
挖宝挂机
  -> 获得未知古物 / 装备 / 材料 / 遭遇怪物
  -> 鉴宝估价：出售、收藏、修复、炼化
  -> 打怪推图：小怪、精英、首领
  -> 获得灵石、经验、装备、古宝碎片
  -> 强化境界、装备、法宝、图鉴
  -> 解锁更高级地图和更高价值古物
```

### 10.2 鉴宝估价界面

界面目标是让玩家感受到“挖到东西以后可能发财”。它需要比普通背包更有仪式感，但操作仍然要轻。

| 区域 | 内容 | 交互 |
| --- | --- | --- |
| 古物展示区 | 古物大图、品质光效、破损状态、未知标签 | 点击查看详情和来源 |
| 线索区 | 年代、材质、灵气、完整度、疑似品阶 | 未鉴定时只显示模糊区间 |
| 估价区 | 估价范围、市场热度、最终成交价 | 鉴定后显示确定价值 |
| 操作区 | 鉴定、出售、收藏、修复、炼化 | 根据古物状态解锁按钮 |
| 记录区 | 最近高价成交、捡漏记录、图鉴进度 | 强化稀有掉落反馈 |

### 10.3 鉴宝估价规则

| 规则 | 说明 |
| --- | --- |
| 未鉴定状态 | 古物刚产出时只显示“疑似品质”和估价范围 |
| 鉴定消耗 | 消耗鉴宝符、灵石或免费鉴定次数 |
| 估价波动 | 同一模板按品质、完整度、灵气值生成最终价格 |
| 捡漏事件 | 低线索古物有小概率鉴定为高价值古物 |
| 赝品事件 | 高线索古物有小概率鉴定为低价值赝品 |
| 修复系统 | 残缺古物可消耗材料修复，提高收藏属性或售价 |
| 收藏图鉴 | 首次收藏某类古物可激活永久属性加成 |
| 炼化回收 | 低价值古物可炼化为强化材料，避免背包堆积 |

估价公式建议：

```text
基础估价 = 古物模板价值 × 品质系数 × 地图系数
完整度影响 = 0.4 + 完整度百分比 × 0.8
灵气影响 = 1 + 灵气评分 / 1000
市场波动 = random(0.85, 1.15)
最终估价 = 基础估价 × 完整度影响 × 灵气影响 × 市场波动
```

### 10.5 打怪界面

打怪采用轻量自动回合制，不做复杂手操。玩家主要通过境界、装备、法宝、图鉴和阵容克制来提升胜率。

| 区域 | 内容 | 交互 |
| --- | --- | --- |
| 敌人信息 | 名称、等级、血条、特性、掉落预览 | 点击查看怪物弱点 |
| 我方信息 | 主角、守护、法宝、战力、属性 | 可进入调整装备/法宝 |
| 战斗区 | 双方站位、技能特效、伤害数字 | 自动播放，可加速 |
| 日志区 | 回合记录、暴击、闪避、控制、掉落 | 可折叠 |
| 操作区 | 挑战、扫荡、调整、退出 | 失败后给提升建议 |

### 10.6 怪物类型

| 类型 | 作用 | 掉落 |
| --- | --- | --- |
| 小怪 | 日常材料和地图进度 | 灵石、经验、普通材料 |
| 精英 | 阶段挑战和稀有掉落 | 装备、鉴宝符、古物碎片 |
| 首领 | 地图解锁和主线推进 | 高品质装备、法宝碎片、稀有古物 |
| 事件怪 | 秘境和限时事件 | 特殊材料、称号、图鉴项 |

### 10.7 怪物特性与克制

| 特性 | 表现 | 克制方式 |
| --- | --- | --- |
| 物抗 | 降低普通攻击和物理技能伤害 | 使用法术法宝、灵气类词条 |
| 法抗 | 降低法术技能伤害 | 使用物理装备、破甲词条 |
| 高闪避 | 容易躲避攻击 | 提升命中、使用必中技能 |
| 毒伤 | 持续扣血 | 提升抗性、使用净化法宝 |
| 控制 | 眩晕、沉默、减速 | 提升控制抵抗 |
| 高爆发 | 前几回合伤害高 | 提升生命、防御、护盾 |

战斗失败时，界面必须给出明确建议：

- 境界不足：提示突破境界。
- 输出不足：提示强化武器或更换克制法宝。
- 生存不足：提示强化防具、提升生命或防御。
- 命中不足：提示使用命中词条或必中技能。
- 地图难度过高：提示回到上一地图刷装备。

### 10.8 战斗结算规则

```text
战斗力 = 主角属性 + 装备属性 + 法宝属性 + 图鉴加成 + 境界加成
伤害 = 攻击 × 技能倍率 × 克制系数 × 暴击系数 - 防御减免
胜利条件 = 敌方生命降为 0
失败条件 = 我方生命降为 0 或超过最大回合数
扫荡条件 = 该关卡首次通关且战力达到推荐战力
```

结算产出：

- 胜利：地图进度、经验、灵石、掉落物。
- 失败：少量经验或无奖励，并给成长建议。
- 首通：额外宝箱、地图解锁、主线任务进度。
- 扫荡：快速获得已通关关卡基础奖励，但稀有奖励概率可略低。

### 10.9 鉴宝与打怪联动

| 联动点 | 说明 |
| --- | --- |
| 打怪掉古物 | 精英和首领可掉落未鉴定古物 |
| 鉴宝给图鉴 | 收藏高价值古物解锁永久属性 |
| 图鉴反哺战斗 | 图鉴提供攻击、防御、生命、挖宝效率等加成 |
| 战斗解锁地图 | 击败首领后解锁更高级挖宝地图 |
| 地图影响估价 | 高级地图产出的古物基础价值更高 |

### 10.12 MVP 验收标准

- 玩家能从挖宝获得未鉴定古物。
- 玩家能在鉴宝界面完成鉴定、出售、收藏、炼化。
- 收藏古物后能提升图鉴进度，并对角色属性生效。
- 玩家能挑战小怪、精英、首领，并看到清晰的战斗过程。
- 首领通关能解锁下一张地图或推进主线。
- 战斗失败时能给出明确成长建议。
- 古物估价、战斗奖励、图鉴加成均由服务端校验。
- 所有古物和战斗奖励都写入流水，重复请求不会重复发奖。

### 10.13 鉴宝界面状态流

鉴宝界面需要把“未知感”和“发财反馈”做清楚，所有按钮都应根据古物状态自动启用或禁用。

```text
未鉴定
  -> 鉴定中
  -> 已鉴定
      -> 出售
      -> 收藏
      -> 修复
      -> 炼化
```

| 状态 | 显示内容 | 可用操作 | 退出后保留 |
| --- | --- | --- | --- |
| 未鉴定 | 疑似品阶、估价区间、来源地图、线索 | 鉴定、返回 | 是 |
| 鉴定中 | 鉴定动画、灵气波动、结果预告 | 跳过动画 | 否 |
| 已鉴定 | 真实名称、真实品质、最终估价、图鉴状态 | 出售、收藏、修复、炼化 | 是 |
| 已出售 | 售价、获得灵石、流水编号 | 查看记录 | 是 |
| 已收藏 | 图鉴进度、永久属性、收藏编号 | 查看图鉴 | 是 |
| 已炼化 | 获得材料、炼化来源 | 查看材料 | 是 |

按钮规则：

- `鉴定`：仅未鉴定古物可用，资源不足时按钮置灰并展示缺少的资源。
- `出售`：已鉴定且未收藏的古物可用；收藏后的古物默认不可出售，避免误操作。
- `收藏`：该古物模板未激活图鉴时优先高亮；重复收藏时只给图鉴经验或材料。
- `修复`：完整度低于 100 且配置允许修复时可用。
- `炼化`：低价值古物默认推荐炼化，高品质古物点击炼化需二次确认。

### 10.14 鉴宝结果表现

鉴宝结果必须分层表现，避免所有古物都像普通道具一样平。

| 结果类型 | 触发条件 | 表现 |
| --- | --- | --- |
| 普通成交 | 常规估价 | 普通结算音效、显示售价 |
| 高价成交 | 最终估价高于模板均值 120% | 强光效、飘字“高价成交” |
| 捡漏 | 疑似品质低但真实品质高 | 特殊提示“捡漏成功” |
| 赝品 | 疑似品质高但真实价值低 | 弱化表现，补偿少量材料 |
| 稀世珍品 | 品质达到最高档或触发保底 | 全屏展示、记录到最近高价 |
| 图鉴激活 | 首次收藏该古物 | 展示永久属性增加 |

推荐结算文案：

- `鉴定完成：获得「青铜古镜」，估价 2,880 灵石。`
- `捡漏成功：这件古物真实品阶高于线索判断。`
- `图鉴激活：攻击 +12，挖宝效率 +0.5%。`
- `该古物疑似赝品，建议炼化为修复材料。`

### 10.15 打怪界面状态流

打怪界面要减少手操压力，核心是“看得懂为什么赢/为什么输”。

```text
关卡选择
  -> 敌人预览
  -> 阵容/装备确认
  -> 自动战斗
  -> 胜利/失败结算
  -> 下一关/扫荡/返回养成
```

| 状态 | 显示内容 | 可用操作 |
| --- | --- | --- |
| 关卡选择 | 地图进度、小怪/精英/首领节点 | 挑战、扫荡、查看掉落 |
| 敌人预览 | 怪物等级、特性、推荐战力、弱点 | 挑战、调整装备 |
| 战斗中 | 血条、技能、伤害、回合数、战斗日志 | 加速、跳过已通关战斗 |
| 胜利 | 掉落、经验、地图进度、首通奖励 | 下一关、扫荡、返回 |
| 失败 | 失败原因、推荐提升项、差距数值 | 去强化、换法宝、返回 |

失败原因判定优先级：

1. 生存不足：我方死亡回合小于最大回合的 50%。
2. 输出不足：达到最大回合但敌方剩余生命高于 20%。
3. 命中不足：未命中次数占攻击次数超过 25%。
4. 克制错误：敌人有物抗/法抗，而玩家主要输出类型被克制。
5. 综合战力不足：推荐战力差距超过 15%。

### 10.16 红点与引导规则

| 入口 | 红点条件 | 消失条件 |
| --- | --- | --- |
| 鉴宝 | 有未鉴定古物、免费鉴宝次数可用 | 全部查看或鉴定完成 |
| 图鉴 | 有可激活收藏、有图鉴奖励可领 | 激活或领取后 |
| 打怪 | 当前战力达到下一关推荐战力、首领可挑战 | 挑战后或战力不足 |
| 装备 | 有更高评分装备、可强化装备 | 查看推荐或完成强化 |
| 境界 | 突破材料足够 | 突破或材料不足 |
| 任务 | 有任务奖励可领 | 领取后 |

新手引导顺序：

1. 第一次挖宝。
2. 第一次获得未知古物。
3. 第一次鉴宝并出售。
4. 第一次装备穿戴。
5. 第一次打怪。
6. 第一次失败后的强化建议。
7. 第一次境界突破。
8. 第一次离线收益领取。

## 11. MVP 研发范围

MVP 目标不是做完整商业版本，而是在 6-10 周内完成一个可测试、可调数值、可接轻付费的闭环版本。首版重点验证三件事：

1. 玩家是否愿意每天回来领取收益、升级、解锁地图。
2. 鉴宝估价、打怪、掉落、装备、境界突破是否有足够爽感和长期追求。
3. 月卡、礼包、外观和加速道具是否能自然嵌入核心循环。

### 11.1 MVP 必做功能

| 模块 | 范围 | 验收标准 |
| --- | --- | --- |

### 11.2 MVP 暂缓功能

| 功能 | 暂缓原因 | 后续时机 |
| --- | --- | --- |
| PVP / 竞技场 | 数值平衡复杂，容易拖慢首版验证 | 核心留存稳定后 |

| 大型赛季活动 | 依赖运营后台和内容产能 | 上线前 1-2 个月 |
| 广告激励 | 会引入 SDK、合规和奖励经济复杂度 | 核心循环验证稳定后 |

## 12. 技术架构落地

### 12.1 总体架构

```text
Unity Client
  -> API Gateway
  -> Game API Service
      -> PostgreSQL/MySQL
      -> Redis
      -> Config Service
      -> Payment Callback Service
      -> Analytics/Event Collector
  -> Admin Console
```

MVP 阶段可以先做单体后端，但模块边界要清晰，避免后续拆分困难。推荐把账号、资产、挂机、掉落、订单、配置、邮件、埋点按目录隔离。

### 12.2 客户端模块

| 模块 | 职责 |
| --- | --- |
| `Core` | 启动流程、状态机、时间同步、错误处理 |
| `Config` | 配置下载、版本校验、本地缓存、配置解析 |
| `Network` | API 封装、重试、鉴权、错误码处理 |
| `Account` | 游客登录、绑定、token 管理 |
| `Inventory` | 背包展示、资产变更提示、道具详情 |
| `Digging` | 当前地图、挂机状态、收益预览、领取表现 |
| `Antique` | 古物列表、鉴宝估价、出售、收藏、修复、炼化 |
| `Battle` | 怪物展示、自动战斗、扫荡、战斗结算 |
| `Equipment` | 装备列表、穿戴、替换、回收、战力刷新 |
| `Quest` | 主线、日常、成就进度和奖励领取 |
| `Shop` | 月卡、礼包、加速道具、订单状态 |
| `UI` | 页面栈、弹窗队列、红点、通用列表 |
| `Analytics` | 埋点队列、失败缓存、批量上报 |

客户端原则：

- 关键资产以服务端返回为准，客户端只做表现和缓存。
- 每次进入主界面后请求一次 `sync`，修正配置、服务器时间、资产摘要。
- UI 页面不直接改资产，统一走本地状态仓库和服务端回包刷新。
- 弱网下允许重试领取，但服务端必须幂等。

### 12.5 鉴宝 API 示例

`POST /antique/appraise`

请求：

```json
{
  "antique_uid": "antique_10001",
  "cost_item_id": "appraisal_token",
  "request_id": "req_20260627_0001"
}
```

响应：

```json
{
  "antique": {
    "uid": "antique_10001",
    "template_id": "bronze_mirror_001",
    "state": "appraised",
    "real_quality": "epic",
    "integrity": 78,
    "aura": 320,
    "final_price": 2880,
    "result_type": "bargain"
  },
  "cost": [
    { "item_id": "appraisal_token", "amount": 1 }
  ],
  "updated_assets": {
    "appraisal_token": 2
  }
}
```

服务端校验：

- 古物必须属于当前玩家。
- 古物状态必须是 `unidentified`。
- `request_id` 不可重复。
- 消耗道具或免费次数必须足够。
- 鉴定结果一旦生成不可被客户端重刷。

### 12.6 战斗 API 示例

`POST /battle/start`

请求：

```json
{
  "stage_id": "stage_010",
  "formation_id": "default",
  "request_id": "req_20260627_0002"
}
```

响应：

```json
{
  "battle_id": "battle_90001",
  "seed": "server_seed_abc",
  "stage_id": "stage_010",
  "player_snapshot": {
    "power": 1820,
    "atk": 280,
    "def": 95,
    "hp": 2400
  },
  "enemy_snapshot": {
    "monster_id": "boss_mine_001",
    "hp": 3200,
    "traits": ["burst", "poison"]
  }
}
```

`POST /battle/finish`

请求：

```json
{
  "battle_id": "battle_90001",
  "client_result": "win",
  "round_count": 8,
  "request_id": "req_20260627_0003"
}
```

响应：

```json
{
  "verified_result": "win",
  "stage_progress": {
    "stage_id": "stage_010",
    "best_result": "win",
    "sweep_unlocked": true
  },
  "rewards": [
    { "item_id": "spirit_stone", "amount": 1200 },
    { "item_id": "appraisal_token", "amount": 1 }
  ],
  "unlock": {
    "map_id": "map_02"
  }
}
```

服务端校验：

- 战斗用服务端快照和种子复算结果，不能完全相信客户端。
- `battle_id` 只能结算一次。
- 失败也写入战斗日志，用于分析关卡难度。
- 首通奖励只能领取一次。

### 12.7 通用错误码

| 错误码 | 含义 | 客户端处理 |
| --- | --- | --- |
| `AUTH_EXPIRED` | 登录过期 | 重新登录 |
| `CONFIG_OUTDATED` | 配置版本过旧 | 拉取最新配置后重试 |
| `ASSET_NOT_ENOUGH` | 资源不足 | 提示缺少资源 |
| `LIMIT_REACHED` | 次数/上限已达 | 提示明日刷新或购买权益 |
| `REWARD_ALREADY_CLAIMED` | 奖励已领取 | 刷新状态 |
| `ORDER_PENDING` | 订单处理中 | 展示等待状态并允许补单 |
| `ORDER_DUPLICATED` | 重复订单 | 查询订单结果 |
| `PLAYER_BANNED` | 账号封禁 | 展示客服提示 |
| `ANTIQUE_STATE_INVALID` | 古物状态不允许当前操作 | 刷新古物列表 |
| `BATTLE_ALREADY_FINISHED` | 战斗已结算 | 刷新关卡进度 |
| `STAGE_LOCKED` | 关卡未解锁 | 展示解锁条件 |

### 13.3 资产流水规则

- 所有货币、材料、装备增减都必须写入流水。
- `request_id` 必须唯一，避免网络重试造成重复发奖。
- 流水 `reason` 使用枚举，例如 `idle_claim`、`quest_reward`、`equipment_recycle`、`iap_purchase`、`admin_mail`。
- 支付、鉴宝、战斗、邮件附件这类奖励必须先校验状态，再发奖，再更新领取状态。

### 14.2 数值曲线原则

- 前 10 分钟每 1-2 分钟给一次明显成长。
- 前 30 分钟至少解锁 2 张地图、完成 1 次境界突破、获得 1 件高品质装备。
- 前 30 分钟至少完成 2 次鉴宝、1 次精英怪战斗、1 次首领战斗。
- 第 1 天目标时长 20-40 分钟，离线回流收益要足够推动下一次升级。
- 第 3 天开始放慢境界速度，把追求转向装备词条、法宝碎片和日常活动。
- 免费玩家每天可完成 70%-80% 的基础成长，月卡和礼包提供效率提升。

### 14.3 掉落模拟要求

研发期需要提供一个简单脚本或后台工具，用于模拟：

- 单地图挂机 1 小时、8 小时、24 小时的资源产出。
- 稀有装备 1000 次、10000 次掉落分布。
- 普通玩家、月卡玩家、礼包玩家的成长差距。
- 古物 1000 次鉴定的估价分布、赝品比例、捡漏比例。
- 小怪、精英、首领的平均通关战力和失败率。
- 每日资源产出与核心消耗是否平衡。

## 19. 风险清单

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 数值膨胀过快 | 生命周期缩短 | 做经济模拟与每日产消看板 |
| 纯本地存档作弊 | 排行榜与付费受损 | 服务端校验关键资产 |
| 内容消耗过快 | 中后期流失 | 周活动、赛季、装备词条追求 |
| 付费强度过高 | 口碑下降 | 控制付费效率上限，增加外观付费 |
| 后端边界不清 | 后续接支付、活动、风控困难 | MVP 就建立资产流水和幂等规则 |
| 配置不可回滚 | 错误数值可能导致线上事故 | 配置发布必须支持校验、灰度、回滚 |
| 首日体验太慢 | 新用户无法感知成长爽感 | 前 30 分钟设定密集目标和高反馈奖励 |
| 估价随机过大 | 玩家觉得不公平 | 明示估价区间，控制赝品概率和保底 |
| 战斗黑盒过强 | 玩家不知道怎么变强 | 失败后给明确克制和养成建议 |
